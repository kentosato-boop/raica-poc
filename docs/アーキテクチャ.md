# RAiCA 2.0 アーキテクチャ

## データフロー

```text
Porters API ──> Sync Adapter ──> PostgreSQL / SQLite
                                      │
React UI <──REST/OpenAPI──> FastAPI ──┼──> Matching Service
                                      ├──> Audit Log
                                      └──> Outbox ──> Gmail / Zalo / Asana Gateway
```

## DB境界

- `candidates`, `companies`, `jobs`: Porters同期対象の正本ミラー
- `matches`: 求人×候補者の計算結果と根拠
- `applications`: 推薦後の選考ステージ
- `action_items`: RA/CAが次に動くキュー
- `contact_logs`: HITL承認を含む接触履歴
- `integration_connections`, `sync_runs`: 外部接続と同期監視
- `outbox_events`: 外部API障害時にも送信指示を失わない再送キュー
- `audit_logs`: 推薦、同期、送信、状態変更の証跡

## 外部連携の原則

1. PortersレスポンスはアダプターでRAiCA形式へ正規化する。
2. 外部送信はDBトランザクション内でOutboxへ記録してから実行する。
3. 接続情報がない場合は成功扱いにせず`not_configured`を返す。
4. 送信失敗は`failed`として残し、UIとAPIから再試行できる。
5. 氏名を含む送信は`human_approved_by`を必須にする。

## 本番移行

- `RAICA_DATABASE_URL`をPostgreSQLへ変更
- Alembicでマイグレーションを適用
- `RAICA_API_KEY`または上位のOIDCゲートウェイを有効化
- Porters契約プランに合わせて候補者・求人エンドポイントを設定
- Gmail/Zalo/Asanaは各社の認証処理を持つ社内Gateway URLへ接続
